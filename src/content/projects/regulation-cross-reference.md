## 課題

大学には就業規則、旅費規程、個人情報保護規程など数十本の規程がある。ある規程を改正したとき、その条文を参照している別の規程も修正が必要になることがあるが、規程間の参照関係を把握しきれず、改正漏れが発生しやすい。改正のたびに全規程を目視で確認するのは現実的ではなく、見落としが起きても気づくのは問題が発覚してからになることが多い。

## この記事が役立つケース

**導入が効果的な場合:**
- 管理している規程が数十本以上あり、規程間の参照関係が複雑になっている
- 年に複数回の規程改正があり、そのたびに影響範囲の確認に時間がかかっている
- 過去に改正漏れが発生し、実務に影響が出たことがある

**現状のままで十分な場合:**
- 規程の数が十数本程度で、担当者が全体を把握できている
- 規程間の参照がほとんどなく、独立した内容の規程が多い
- 改正頻度が年に1〜2回程度で、時間をかけて確認できる

## 解決策

すべての規程を **NotebookLM** に登録し、改正のたびに NotebookLM に改正内容を伝えて影響範囲を確認する。NotebookLM は全規程の本文をソースとして保持しているため、「○○規程第△条に定める」のような明示的な参照だけでなく、同じ用語を前提としている条文や類似の規定など、暗黙的な関連も検出できる。

スプレッドシートによる参照関係マップの構築・維持は不要。NotebookLM 自体が全規程の内容を把握しており、改正のたびに必要な情報をその場で分析できるため、別途マップを管理する手間をかける必要がない。

## 仕組みの全体像

```
【初回のみ】NotebookLM の準備（30分〜1時間）
  全規程を NotebookLM に登録する

【改正のたびに】影響範囲のチェック（10〜20分）
  NotebookLM に改正内容を伝える
    → 影響を受ける規程・条文の一覧を取得
      → 連鎖的な影響を同じ会話で確認
        → 改正案を作成

【規程の追加・廃止・改正後】ソースの更新（数分）
  NotebookLM のソースを差し替える
```

## 前提条件

- Google Workspace Business Standard 以上、または NotebookLM Plus の有料プラン
- NotebookLM Plus では1ノートブックあたり最大300ソースまで登録できるため、規程数が数十〜百本程度であれば十分に収まる

## 構築手順

### 1. 規程ファイルの準備

すべての規程を Google ドキュメントまたは PDF で用意する。ファイル名は規程名と一致させておく（例: 「就業規則.pdf」「旅費規程.pdf」）。

**ポイント:**
- 紙しかない規程は、Google ドライブにスキャンしてアップロードすると OCR で文字認識される
- Google ドライブの1つのフォルダにまとめておくと、NotebookLM への登録やファイルの差し替えが楽になる

### 2. NotebookLM にすべての規程を登録する

1. [notebooklm.google.com](https://notebooklm.google.com/) で新しいノートブックを作成する
2. 「ソースを追加」からすべての規程ファイルを登録する

Google ドライブから直接追加できるため、フォルダにまとめておけば効率的に登録できる。このノートブックは以降ずっと使い回す。

### 3. 初回の動作確認

登録が完了したら、NotebookLM が全規程を正しく読み取れているか確認する。

**確認プロンプト:**

```
登録されているすべての規程の名称を一覧にしてください。
```

想定する規程がすべて含まれているか確認する。欠けているものがあればソースを追加する。

続けて、参照関係を正しく検出できるか試す。

**テストプロンプト:**

```
就業規則を参照している他の規程・条文を
すべて洗い出してください。

「就業規則第○条に定める」のような明示的な参照だけでなく、
就業規則で定義されている用語を使用している条文も
含めて確認してください。
```

出力結果を、就業規則を実際に参照していることがわかっている規程と突き合わせて、精度を確認する。

## 改正時の使い方

### 1. 影響範囲を確認する

規程を改正する際、NotebookLM のノートブックを開いて改正内容を伝える。

**プロンプト:**

```
就業規則の第15条（出張の定義）を以下のとおり改正します。

【現行】
出張とは、用務のため勤務地を離れて旅行することをいう。

【改正案】
出張とは、用務のため通常の勤務場所を離れて旅行することをいう。
ただし、テレワーク勤務地からの旅行を含む。

この改正により影響を受ける可能性のある規程・条文を
すべて洗い出してください。

以下の観点で確認してください。
- 就業規則第15条を直接参照している条文
- 「出張」の定義を前提としている条文
- 出張に関連する手続き・手当・費用を定めている条文

影響がある場合、それぞれについて以下を示してください。
- 影響を受ける規程名と条文
- 影響の内容（修正が必要か、確認のみでよいか）
- 修正が必要な場合の改正案
```

**NotebookLM の出力例:**

```
【直接的な参照】
1. 旅費規程 第3条
   - 「就業規則第15条に定める出張」を引用
   - テレワーク勤務地からの出張が追加されるため、
     旅費の起算地に関する規定の修正が必要

【間接的な関連】
2. テレワーク規程 第8条
   - 「出張」の文言を使用しているが、就業規則の参照はない
   - テレワーク勤務地からの出張を含む旨の整合性確認が必要

3. 労災補償規程 第4条
   - 「出張中の事故」に関する規定あり
   - テレワーク勤務地からの出張も対象になるか確認が必要
```

### 2. 連鎖的な影響を確認する

修正が必要な規程が見つかった場合、その修正がさらに他の規程に影響する可能性がある。**同じ会話の中で続けて確認する。**

**プロンプト:**

```
旅費規程 第3条を以下のとおり修正する場合、
さらに影響を受ける規程・条文がないか確認してください。

【旅費規程 第3条の修正内容】
（修正内容を記載）
```

新たに修正が必要な規程が見つかった場合は、同様に確認を繰り返す。修正が必要な規程がなくなった時点で確認は完了。通常は1〜2回で連鎖は止まるが、基幹的な規程（就業規則、経理規程など）の改正では3段階以上になることもある。

### 3. 改正後にソースを更新する

改正が施行されたら、NotebookLM のソースを最新版に差し替える。

**手順:**
1. 改正後の規程ファイルを Google ドライブに保存する（改正前のファイルを上書き、または新しいファイルに差し替え）
2. NotebookLM のソースで、改正前の規程を削除し、改正後の規程を追加する

Google ドライブのファイルを直接上書きした場合は、NotebookLM 側でソースの再同期を行えば自動的に最新版に更新される。

規程を新設した場合はソースを追加、廃止した場合はソースを削除する。

## 参照関係の全体像を把握したい場合（任意）

NotebookLM による改正チェックとは別に、「規程間の参照関係を一覧で俯瞰したい」場合は、NotebookLM に参照関係の一覧を生成させることができる。

**プロンプト:**

```
登録されているすべての規程について、
他の規程や条文を参照している箇所を抽出し、
以下の形式で一覧にしてください。

| 参照元の規程名 | 参照元の条文 | 参照先の規程名 | 参照先の条文 | 参照内容 |
```

この出力をスプレッドシートに貼り付ければ、参照関係の全体像を一覧で確認できる。ただし、このスプレッドシートは改正時の影響チェックには使わない（NotebookLM に直接確認した方が正確で早い）。規程体系全体の構造を把握したいときや、管理職への説明資料として使う程度の位置づけでよい。

## 担当者が異動した場合

規程管理の担当者が異動する際は、以下を引き継ぐ。

- NotebookLM のノートブック（全規程がソースとして登録済み）
- 規程ファイルの保管場所（Google ドライブのフォルダ）
- この記事の手順

NotebookLM のノートブックが引き継がれれば、新任者でも改正時の影響確認を同じ精度で実施できる。属人的な知識への依存を減らすことが、この仕組みの最大の利点でもある。

## 導入効果

| 項目 | 導入前 | 導入後 |
|------|--------|--------|
| 初期準備 | — | 全規程を NotebookLM に登録（30分〜1時間） |
| 改正時の影響確認 | 全規程を目視で確認（数時間〜数日） | NotebookLM に質問（10〜20分） |
| 改正後の後処理 | — | ソースの差し替え（数分） |
| 改正漏れのリスク | 参照関係を見落としやすい | 明示的参照＋暗黙的関連を AI が検出 |
| 暗黙的な関連の発見 | 担当者の経験に依存 | NotebookLM が全文から検出 |
| 引き継ぎ | 担当者の記憶に依存 | ノートブックを共有するだけ |

## 応用例

- **条例・要綱の管理**: 自治体の条例・要綱間の参照関係管理にも同じ手法が使える
- **契約書の管理**: 関連する複数の契約書間の参照関係を管理し、一方の変更時に他方への影響を確認する
- **マニュアル体系の管理**: 業務マニュアル間の参照関係を管理し、手続き変更時の改訂漏れを防ぐ

## この方式の限界

NotebookLM は万能ではない。以下の限界を理解した上で運用する。

- **AI の出力は必ず人間が確認する。** 特に法的な影響がある規程改正では、最終判断は必ず担当者が行う
- NotebookLM が影響を「検出しなかった」ことは、「影響がない」ことの保証にはならない。重要な改正では、NotebookLM の結果に加えて、関連が深い規程については目視でも確認する
- 規程の新設・廃止があった場合は、NotebookLM のソースの追加・削除を忘れずに行う。ソースが古いまま放置すると、影響チェックの精度が下がる

## 注意点

- 規程ファイルに個人情報や機密情報が含まれる場合は、NotebookLM のデータ取り扱いポリシーを確認した上で利用する。Google Workspace 版の NotebookLM はエンタープライズ級のデータ保護が適用される
- 規程ファイルは Google ドライブの1つのフォルダで管理し、常に最新版を維持する。ソースの差し替え忘れを防ぐため、改正の決裁・施行のチェックリストに「NotebookLM のソース更新」を含めておく
