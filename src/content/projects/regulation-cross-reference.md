## 課題

大学には就業規則、旅費規程、個人情報保護規程など数十本の規程がある。ある規程を改正したとき、その条文を参照している別の規程も修正が必要になることがあるが、規程間の参照関係を把握しきれず、改正漏れが発生しやすい。改正のたびに全規程を目視で確認するのは現実的ではなく、見落としが起きても気づくのは問題が発覚してからになることが多い。

## この記事が役立つケース

**導入が効果的な場合:**
- 管理している規程が数十本以上あり、規程間の参照関係が複雑になっている
- 年に複数回の規程改正があり、そのたびに影響範囲の確認に時間がかかっている
- 過去に改正漏れが発生し、実務に影響が出たことがある

**現状のままで十分な場合:**
- 規程の数が十数本程度で、担当者が全体を把握できている
- 規程間の参照がほとんどなく、独立した内容の規程が多い
- 改正頻度が年に1〜2回程度で、時間をかけて確認できる

## 解決策

すべての規程を **Gemini** に読み込ませ、規程間の参照関係（「○○規程第△条に定める」などの記述）を自動で抽出し、**Google スプレッドシート** で参照関係マップとして管理する。改正時には **NotebookLM** に参照関係マップと改正内容を渡し、影響範囲を確認する。

## 仕組みの全体像

```
【初回のみ】参照関係マップの構築
  全規程を Gemini に添付
    → 規程間の参照関係を抽出
      → 主要規程で抜き取り検証（漏れがあれば再抽出）
        → スプレッドシートで一覧管理
          → 曖昧な参照表現も別途リスト化

【改正のたびに】影響範囲のチェック
  NotebookLM に参照関係マップ＋全規程＋改正内容を渡す
    → 影響を受ける規程・条文の一覧を取得
      → 修正が必要な規程について連鎖的な影響を再確認
        → 改正案を作成
          → 改正後に参照関係マップを更新

【年1回】マップの再構築
  全規程を Gemini で再抽出
    → 既存マップとの差分を確認・更新
```

## 構築手順

### 1. 規程ファイルの準備

すべての規程を Google ドキュメントまたは PDF で用意する。ファイル名は規程名と一致させておく（例: 「就業規則.pdf」「旅費規程.pdf」）。

**ポイント:**
- 紙しかない規程は、Google ドライブにスキャンしてアップロードすると OCR で文字認識される
- 規程の数が多い場合は、Google ドライブの1つのフォルダにまとめておく

### 2. Gemini で参照関係を抽出する

[gemini.google.com](https://gemini.google.com/) に規程ファイルを添付し、参照関係を抽出する。一度に添付できるファイル数には上限があるため、規程の数が多い場合は数回に分けて処理する。

**プロンプト:**

```
添付したすべての規程について、
他の規程や条文を参照している箇所を抽出し、
以下の形式で一覧にしてください。

| 参照元の規程名 | 参照元の条文 | 参照先の規程名 | 参照先の条文 | 参照内容 |

「○○規程第△条に定める」「○○規則の規定に基づき」
「○○規程を準用する」などの表現を漏れなく抽出してください。

条文番号が明示されていない場合（規程名のみの参照）は、
参照先の条文欄に「規程全体」と記載してください。
```

**出力例:**

| 参照元の規程名 | 参照元の条文 | 参照先の規程名 | 参照先の条文 | 参照内容 |
|--------------|------------|--------------|------------|---------|
| 旅費規程 | 第3条 | 就業規則 | 第15条 | 出張の定義 |
| 旅費規程 | 第10条 | 経理規程 | 第8条 | 精算手続き |
| 個人情報保護規程 | 第5条 | 就業規則 | 第40条 | 懲戒処分 |
| 研究費取扱規程 | 第12条 | 経理規程 | 規程全体 | 経理処理の準用 |

### 3. 抽出結果を検証する

Gemini の抽出には漏れが起きうる。特に以下のような表現は見落としやすいため、初回は必ず人間が検証する。

**見落としやすい参照表現:**
- 「前条の例による」「前項を準用する」（規程名が書かれていない）
- 「本学の規程に基づき」「別に定める」（参照先が特定できない）
- 「第○条から第○条までの規定にかかわらず」（除外条件としての参照）

**検証方法:**

参照されている回数が多いと予想される主要規程（就業規則、経理規程など）5〜6本について、目視で参照箇所を確認し、Gemini の抽出結果と突き合わせる。全規程を確認する必要はなく、主要規程の抜き取り検証で抽出の精度を判断できる。

漏れが見つかった場合は、漏れのパターンを Gemini に伝えて再抽出する。

**再抽出のプロンプト例:**

```
前回の抽出では以下のような参照表現が漏れていました。
- 「前条の例による」
- 「別に定める」
- 「第○条の規定にかかわらず」

これらの表現を含め、添付した規程をもう一度確認し、
漏れていた参照関係を追加で抽出してください。
```

### 4. 参照関係マップをスプレッドシートで管理する

検証済みの抽出結果を Google スプレッドシートに転記し、参照関係マップとして管理する。

**スプレッドシートの構成:**

#### 参照関係シート

Gemini の出力をそのまま管理する。フィルタを設定しておくと、特定の規程を改正する際に「この規程を参照している他の規程」を即座に絞り込める。

**フィルタの使い方:**
- 「参照先の規程名」列でフィルタ → 改正する規程を選択 → 影響を受ける規程の一覧が表示される
- 例: 就業規則を改正する場合、「参照先の規程名」を「就業規則」でフィルタすると、就業規則を参照しているすべての規程・条文が表示される

#### 曖昧な参照リスト

「別に定める」「本学の規程に基づき」など、参照先が特定できない表現を別シートでリスト化する。

| 規程名 | 条文 | 記述内容 | 推定される参照先 | 備考 |
|--------|------|---------|---------------|------|
| 旅費規程 | 第15条 | 「別に定める」 | 旅費支給細則（未制定？） | 要確認 |
| 研究費取扱規程 | 第3条 | 「本学の規程に基づき」 | 経理規程と推定 | |

改正時にこのリストも確認対象にする。「別に定める」とされている細則が実際に存在するのか、存在しない場合は規程の文言を修正すべきかの判断材料にもなる。

#### 規程一覧シート

規程の基本情報を管理する。

| 規程名 | 最終改正日 | 条文数 | 参照している規程数 | 参照されている規程数 |
|--------|----------|--------|----------------|----------------|
| 就業規則 | 2025/10/1 | 45 | 2 | 8 |
| 旅費規程 | 2024/4/1 | 20 | 3 | 1 |

「参照されている規程数」が多い規程ほど、改正時の影響範囲が広い。改正の際は特に慎重に確認する必要がある。

### 5. 改正時に NotebookLM で影響範囲を確認する

規程を改正する際、**NotebookLM** に参照関係マップと全規程を読み込ませ、改正内容の影響範囲を確認する。

**NotebookLM を使う理由:** 参照関係マップだけでなく全規程の本文もソースとして読み込ませることで、参照関係マップに記載されていない暗黙的な関連（同じ用語の定義を共有している、類似の条文がある等）も検出できる。

**手順:**

1. NotebookLM で新しいノートブックを作成する
2. 以下をソースとして追加する
   - 参照関係マップ（スプレッドシート）
   - 全規程ファイル（Google ドキュメントまたは PDF）
3. 改正内容を伝えて影響範囲を確認する

**プロンプト:**

```
就業規則の第15条（出張の定義）を以下のとおり改正します。

【現行】
出張とは、用務のため勤務地を離れて旅行することをいう。

【改正案】
出張とは、用務のため通常の勤務場所を離れて旅行することをいう。
ただし、テレワーク勤務地からの旅行を含む。

この改正により影響を受ける可能性のある規程・条文を
すべて洗い出してください。

参照関係マップに記載されている直接的な参照だけでなく、
「出張」の定義を前提としている条文や、
出張に関連する手続きを定めている条文も含めて確認してください。

影響がある場合、それぞれについて以下を示してください。
- 影響を受ける規程名と条文
- 影響の内容（修正が必要か、確認のみでよいか）
- 修正が必要な場合の改正案
```

**NotebookLM の出力例:**

```
【直接的な影響（参照関係マップより）】
1. 旅費規程 第3条
   - 「就業規則第15条に定める出張」を引用
   - テレワーク勤務地からの出張が追加されるため、
     旅費の起算地に関する規定の修正が必要

【間接的な影響】
2. テレワーク規程 第8条
   - 「出張」の文言を使用しているが、就業規則の参照はない
   - テレワーク勤務地からの出張を含む旨の整合性確認が必要

3. 労災補償規程 第4条
   - 「出張中の事故」に関する規定あり
   - テレワーク勤務地からの出張も対象になるか確認が必要
```

### 6. 連鎖的な影響を確認する

規程Aの改正が規程Bの修正を必要とし、その規程Bの修正がさらに規程Cに影響する場合がある。この連鎖を見逃さないために、**修正が必要な規程が見つかるたびに、その規程についても影響範囲を再確認する。**

**手順:**

1. ステップ5で NotebookLM が「修正が必要」と判定した規程を確認する
2. その規程の修正内容について、再度 NotebookLM に影響範囲を確認する

**プロンプト:**

```
先ほどの就業規則の改正に伴い、
旅費規程 第3条を以下のとおり修正する必要があるとのことでした。

【旅費規程 第3条の修正内容】
（修正内容を記載）

この修正により、さらに影響を受ける規程・条文がないか確認してください。
```

3. 新たに修正が必要な規程が見つかった場合は、さらにその規程についても同様に確認する
4. **修正が必要な規程がなくなるまで繰り返す**

通常は1〜2回で連鎖は止まるが、基幹的な規程（就業規則、経理規程など）の改正では3段階以上の連鎖が起きることもある。

### 7. 改正後に参照関係マップを更新する

改正が完了したら、参照関係マップを最新の状態に更新する。

**手順:**

1. 改正した規程を Gemini に添付する
2. 参照関係に変更がないか確認するプロンプトを実行する

**プロンプト:**

```
添付した改正後の規程について、
他の規程や条文を参照している箇所を抽出してください。

以下は改正前の参照関係です。変更点がある場合は明示してください。

（改正前の参照関係をスプレッドシートから貼り付け）
```

3. 変更があればスプレッドシートを更新する
4. 規程一覧シートの「最終改正日」も更新する
5. 曖昧な参照リストに変更があれば、そちらも更新する

## 運用サイクル

```
規程改正の発議
  → スプレッドシートの参照関係マップで影響範囲を簡易確認
    → NotebookLM で詳細な影響範囲を分析
      → 連鎖的な影響がなくなるまで繰り返し確認
        → 改正案の作成（関連規程の修正を含む）
          → 改正の決裁・施行
            → 参照関係マップの更新
```

## 参照関係マップの長期メンテナンス

参照関係マップは作って終わりではなく、精度を維持し続けることが重要。以下の運用で陳腐化を防ぐ。

### 改正のたびに行うこと

- 改正した規程の参照関係をマップに反映する（ステップ7）
- 新設した規程があればマップに追加する
- 廃止した規程があればマップから削除し、その規程を参照していた側の規程も確認する

### 年1回の定期メンテナンス（年度末推奨）

マップの精度を保つため、年に1回、全規程を Gemini に再度読み込ませてマップ全体を再構築する。

**再構築する理由:**
- 改正時の更新漏れが蓄積している可能性がある
- Gemini のモデル更新により、以前は検出できなかった参照表現を拾えるようになっている場合がある
- 年度をまたぐ際に規程の新設・廃止が集中しやすい

**手順:**

1. 全規程の最新版を Gemini に添付し、ステップ2と同じプロンプトで参照関係を抽出する
2. 新しい抽出結果と既存のマップを突き合わせる

**突き合わせのプロンプト:**

```
以下の2つの参照関係一覧を比較してください。

【既存のマップ】
（スプレッドシートから貼り付け）

【新規抽出結果】
（今回の Gemini 出力を貼り付け）

差分を以下の形式で出力してください。
- 既存マップにあるが新規抽出にない項目（削除された参照）
- 新規抽出にあるが既存マップにない項目（新たに発見された参照）
- 内容が変わった項目（条文番号の変更など）
```

3. 差分を確認し、マップを更新する
4. 曖昧な参照リストも同様に見直す

### 担当者が異動した場合

規程管理の担当者が異動する際は、以下を引き継ぐ。

- 参照関係マップのスプレッドシート
- NotebookLM のノートブック（全規程がソースとして登録済み）
- この記事の手順（改正時の確認フローと年次メンテナンスの方法）

参照関係マップとノートブックが引き継がれれば、新任者でも改正時の影響確認を同じ精度で実施できる。属人的な知識への依存を減らすことが、この仕組みの最大の利点でもある。

## 導入効果

| 項目 | 導入前 | 導入後 |
|------|--------|--------|
| 影響範囲の確認 | 全規程を目視で確認（数時間〜数日） | スプレッドシートのフィルタ＋NotebookLM で数分 |
| 改正漏れのリスク | 参照関係を見落としやすい | 参照関係マップで網羅的に把握 |
| 暗黙的な関連の発見 | 担当者の経験に依存 | NotebookLM が全文から検出 |
| 改正履歴の管理 | 紙の決裁文書に依存 | スプレッドシートで一元管理 |

## 応用例

- **条例・要綱の管理**: 自治体の条例・要綱間の参照関係管理にも同じ手法が使える
- **契約書の管理**: 関連する複数の契約書間の参照関係を管理し、一方の変更時に他方への影響を確認する
- **マニュアル体系の管理**: 業務マニュアル間の参照関係を管理し、手続き変更時の改訂漏れを防ぐ

## 注意点

- 参照関係マップはあくまで参照の「記述」を抽出したものであり、暗黙的な依存関係（同じ用語を定義なしに使っている等）はマップだけでは把握できない。改正時の NotebookLM による確認を省略しない
- Gemini や NotebookLM の出力は必ず人間が確認する。特に法的な影響がある規程改正では、最終判断は必ず担当者が行う
- 規程の新設・廃止があった場合は、参照関係マップへの追加・削除を忘れずに行う
