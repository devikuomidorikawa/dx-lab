## 課題

学内の文書管理で、ファイル名の付け方が人によってバラバラで、どこに何があるかわからない状態が続いていた。命名規則を策定しても定着しない。規則を覚えていない、調べるのが面倒、急いでいるときに適当な名前で保存してしまう——結局、ルールがあっても守られなければ意味がない。さらに「最終版」「最終版2」「最終版(修正)」のようなファイルが乱立し、どれが最新かも確定済みかもわからない。

## この記事が役立つケース

**導入が効果的な場合:**
- 命名規則を策定したが定着していない、または策定すること自体が後回しになっている
- 共有フォルダ内でファイルを探すのに時間がかかる状況が慢性化している
- 「最終版」「最終版(修正)」のようなファイルが乱立し、どれが最新かわからない
- 部署異動や担当変更のたびに「あのファイルはどこ？」という問い合わせが発生する

**現状のままで十分な場合:**
- 共有フォルダの利用者が数名で、全員がファイルの場所を把握できている
- 既に命名規則が定着しており、運用に問題がない
- ファイル数が少なく、検索すればすぐに見つかる

## 解決策

命名規則に基づくファイル名と保存先フォルダの両方を **Gemini** の Gem（カスタム AI）が提案する仕組みを構築する。担当者は Gem に「このファイルの名前を付けて」と聞くだけで、命名規則に沿ったファイル名と保存先フォルダが提案される。提案どおりにファイル名を付けて **Google ドライブ** の該当フォルダに保存するだけで、文書が整理される。命名規則を覚える必要はない。

## 仕組みの全体像

```
【ファイルの保存】
  担当者が Gem に「このファイルの名前を付けて」と依頼
    → Gem がファイル名と保存先フォルダを提案
      → 担当者がそのファイル名で指定フォルダに保存

【確定文書の作成】
  文書が承認・決裁されたら
    → Gem に「確定版のファイル名をお願い」と依頼
      → Gem が _確定 付きのファイル名と保存先（確定フォルダ）を提案
        → 担当者が PDF 化して確定フォルダに保存
```

## 前提条件

- Gemini Business 以上（Gem の作成に必要）

## 構築手順

### 1. 既存の文書管理規則の確認

多くの大学には文書管理規程、文書分類基準表、文書保存年限表などが既に存在する。**命名規則やフォルダ構成をゼロから考えるのではなく、まず自学の規則を確認し、それを土台にする。**

**確認すべき規則・資料:**

| 資料名 | 確認する内容 |
|--------|------------|
| 文書管理規程（文書取扱規程） | 文書の分類体系、保存年限、決裁区分 |
| 文書分類基準表 | 文書のカテゴリ分けの基準（→ 種別コードの設計に使う） |
| 文書保存年限表 | 保存期間ごとの分類（→ フォルダ構成の参考にする） |
| 個人情報保護規程 | 個人情報を含む文書の取り扱い |
| 法人文書ファイル管理簿 | 国立大学法人の場合、公文書管理法に基づく管理簿 |

これらの規則が見つからない場合や、紙文書のみを対象とした古い規程しかない場合は、ステップ2で種別コードを新たに設計する。

**Gemini を使った分析:**

規則の量が多い場合は、Gemini に文書分類基準表や保存年限表を読み込ませて整理させる。

**プロンプト:**

```
添付した文書分類基準表をもとに、
Google ドライブのフォルダ構成と種別コードを設計してください。

【要件】
- 種別コードは半角英字の大文字、10文字以内
- フォルダは10個以内にまとめる（細かすぎると使われない）
- 分類基準表のカテゴリが多い場合は、
  類似するものをまとめて1つの種別コードにする
- 結果を表形式で出力する
```

Gemini の出力をそのまま使うのではなく、実際の業務で使いやすいかどうかを担当者が判断して調整する。

### 2. 命名規則の設計

既存の文書管理規則で定められた分類体系をもとに、種別コードとフォルダ構成を決定する。命名規則は「覚えなくても使える」ことが最も重要。Gem が提案するため、人間が規則を暗記する必要はないが、規則自体はシンプルに保つ。

**命名規則の形式:**

```
[種別コード]_[内容]_[日付]

例:
GIJI_教授会_20260305
SHINSEI_出張申請_田中_20260305
KIAN_キャンパス再編_20260305
HOUKOKU_FD活動報告_20260305
TEMPLATE_出張申請書
```

**種別コードの一覧（例）:**

| 種別コード | 内容 | 保存先フォルダ |
|-----------|------|--------------|
| GIJI | 議事録 | 01_会議・議事録 |
| KIAN | 企画書・提案書 | 02_企画・提案 |
| SHINSEI | 申請書・届出 | 03_申請・届出 |
| KITEI | 規程・要綱 | 04_規程・要綱 |
| HOUKOKU | 報告書 | 05_報告書 |
| KEIRI | 経理・予算関連 | 06_経理・予算 |
| TEMPLATE | テンプレート | 07_テンプレート |
| SONOTA | その他 | 99_その他 |

部署の業務に合わせて種別コードを追加・変更する。

#### 文書のライフサイクルによる3分類

すべての文書は、以下の3つのいずれかに分類される。**この分類がバージョン管理の基本方針になる。**

| 分類 | 説明 | 例 | ファイル名での区別 |
|------|------|-----|------------------|
| **確定文書** | 承認・決裁を経て内容が確定し、以降変更されない文書 | 議事録（承認後）、規程（施行後）、決裁済み稟議書、提出済み報告書 | `_確定` を付けて PDF 化 |
| **作業文書** | 作成中・編集中で、いずれ確定する文書 | 議事録（下書き）、企画書（検討中）、申請書（作成中） | 何も付けない（常に最新版） |
| **継続文書** | 「確定」の概念がなく、常に更新され続ける文書 | テンプレート、マニュアル、連絡先一覧、FAQ | 何も付けない（常に最新版） |

**考え方:**
- **作業文書**と**継続文書**は、ファイルが常に1つだけ存在し、それが最新版。過去の編集内容は Google ドライブのバージョン履歴で確認できる
- **確定文書**は、確定した時点で PDF に書き出して別ファイルとして保存する。PDF にすることで「これ以降変更されない」ことが誰の目にも明確になる

#### バージョン管理の方針: ファイル名にバージョンを付けない

`_v1`、`_v2`、`_最終版`、`_最終版(修正)` のようなバージョン表記は**一切使わない**。どれが最新かわからなくなる原因はすべてここにある。

**ルール:**
- 作業中のファイルは常に1つ。上書き保存する（Google ドキュメントなら自動保存）
- 過去の版を確認したい場合は、Google ドライブのバージョン履歴を使う
- 「確定」した文書だけが `_確定.pdf` として別ファイルになる

**バージョン履歴の使い方:**
1. Google ドライブでファイルを右クリック →「版を管理」
2. 過去のバージョン一覧が表示される
3. 任意のバージョンをダウンロード、または現在の版として復元できる

Google ドキュメント・スプレッドシート・スライドは編集のたびに自動でバージョン履歴が記録される。Word・Excel・PDF は「新しいバージョンをアップロード」で履歴を追加できる。

#### 確定文書の作り方

文書が承認・決裁されて確定したら、以下の手順で確定版を作成する。

1. 作業文書を PDF として書き出す
2. ファイル名の末尾に `_確定` を付ける
3. 保存先フォルダの「確定」サブフォルダに保存する

```
作業中:   KIAN_キャンパス再編_20260305          ← Google ドキュメント（編集可能）
確定後:   KIAN_キャンパス再編_20260305_確定.pdf  ← PDF（変更不可）
```

**確定後の作業文書はどうするか:**
- 確定後に元の作業文書を編集する必要がなければ、削除するかアーカイブフォルダに移動する
- 確定文書を元にさらに改訂する可能性がある場合（規程など）は、作業文書をそのまま残しておき、次回の改訂時に編集を再開する

#### 3分類の具体例

```
【確定文書の流れ】
  GIJI_教授会_20260305（作業文書として作成）
    → 内容の確認・承認
      → GIJI_教授会_20260305_確定.pdf（PDF化して確定）
        → 確定フォルダに保存。元の作業文書は削除またはアーカイブ

【作業文書の流れ】
  KIAN_新学部設置_20260305（作成開始）
    → 編集を重ねる（バージョン履歴に自動記録）
      → いずれ確定文書になる、または不採用で削除

【継続文書の流れ】
  TEMPLATE_出張申請書（作成）
    → 必要に応じて更新し続ける（バージョン履歴に自動記録）
      → 確定することはない。常にこのファイルが最新
```

### 3. フォルダ構成の準備

Google ドライブに以下のフォルダを作成する。各種別フォルダの中に「確定」サブフォルダを設け、作業中の文書と確定済みの文書を物理的に分離する。

```
部署共有/
  ├── 01_会議・議事録/
  │   └── 確定/             ← 承認済み議事録（PDF）
  ├── 02_企画・提案/
  │   └── 確定/             ← 決裁済み企画書（PDF）
  ├── 03_申請・届出/
  │   └── 確定/             ← 提出済み申請書（PDF）
  ├── 04_規程・要綱/
  │   └── 確定/             ← 施行済み規程（PDF）
  ├── 05_報告書/
  │   └── 確定/             ← 提出済み報告書（PDF）
  ├── 06_経理・予算/
  │   └── 確定/             ← 決裁済み経理文書（PDF）
  ├── 07_テンプレート/       ← 継続文書のみ（確定フォルダ不要）
  └── 99_その他/
```

**ポイント:**
- 各フォルダを開いたとき、**直下にあるのが作業中の文書**、**「確定」フォルダの中にあるのが確定済みの文書（PDF）**。この構造だけで状態が一目でわかる
- テンプレートフォルダには「確定」サブフォルダを作らない（テンプレートは継続文書であり、確定の概念がない）

### 4. Gem（カスタム AI）の作成

Gemini で命名規則とフォルダ構成を知っている Gem を作成する。担当者はこの Gem にファイルの内容や用途を伝えるだけで、命名規則に沿ったファイル名と保存先フォルダが提案される。

**手順:**
1. [gemini.google.com](https://gemini.google.com/) を開く
2. 左メニューの「Gem マネージャー」→「新しい Gem」を選択
3. 以下の設定を入力する

**Gem の名前:** 文書命名アシスタント

**Gem の指示:**

```
あなたは学内文書のファイル名と保存先フォルダを提案するアシスタントです。
以下の命名規則に従って提案してください。

【命名規則】
形式: [種別コード]_[内容]_[日付]

【種別コードと保存先フォルダ】
- GIJI: 議事録 → 01_会議・議事録
- KIAN: 企画書・提案書 → 02_企画・提案
- SHINSEI: 申請書・届出 → 03_申請・届出
- KITEI: 規程・要綱 → 04_規程・要綱
- HOUKOKU: 報告書 → 05_報告書
- KEIRI: 経理・予算関連 → 06_経理・予算
- TEMPLATE: テンプレート（日付不要） → 07_テンプレート
- SONOTA: 上記に該当しないもの → 99_その他

【日付の形式】
YYYYMMDD（例: 20260305）

【ルール】
- 内容は10文字以内で簡潔に
- スペースは使わない（アンダースコアで区切る）
- ファイルの拡張子は含めない（保存時にユーザーが付ける）
- 複数の候補がある場合は種別コードの選択理由も添える
- バージョン番号（v1, v2）や「最終版」は絶対に付けない。
  バージョン管理は Google ドライブのバージョン履歴で行う

【文書の状態による命名】
- 作業中の文書: そのまま → フォルダ直下に保存
  （例: GIJI_教授会_20260305 → 01_会議・議事録/）
- 確定した文書: 末尾に「_確定」を付ける → 確定フォルダに保存
  （例: GIJI_教授会_20260305_確定 → 01_会議・議事録/確定/）
- 継続文書（テンプレート等）: そのまま。確定にならない

ユーザーが「確定版を作りたい」「承認済みにしたい」と言った場合のみ
「_確定」を付けたファイル名と確定フォルダを提案してください。
通常は「_確定」を付けないファイル名を提案してください。

【応答形式】
以下の形式で提案してください。

ファイル名: XXXXX_XXXXX_XXXXXXXX
保存先: XX_XXXXXXX/

種別コードの判断に迷う場合は、候補を2つ示して
ユーザーに選んでもらってください。
```

4. 「保存」をクリック

**使い方の例:**

担当者が Gem に話しかける:

```
3月5日の教授会の議事録のファイル名をお願いします
```

Gem の応答:

```
ファイル名: GIJI_教授会_20260305
保存先: 01_会議・議事録/
```

確定版を作りたい場合:

```
教授会の議事録が承認されたので確定版のファイル名をお願いします
```

Gem の応答:

```
ファイル名: GIJI_教授会_20260305_確定
保存先: 01_会議・議事録/確定/
※ PDF として保存してください
```

### 5. 動作確認

構築が完了したら、以下のテストを行う。

1. Gem にさまざまな文書のファイル名を提案してもらい、命名規則どおりに出力されるか確認する
2. 作業文書、確定文書、継続文書それぞれのケースで正しい保存先が提案されるか確認する
3. 種別コードの判断が曖昧な文書（例: 会議で使う予算資料）で、Gem が適切に候補を提示するか確認する

## 運用フロー

### 日常の操作（担当者）

1. ファイルを保存するとき、Gem に「○○のファイル名をお願い」と聞く
2. 提案されたファイル名を付け、指定された保存先フォルダに保存する

### 確定文書を作るとき

1. Gem に「○○の確定版のファイル名をお願い」と聞く
2. 作業文書を PDF として書き出す
3. 提案された `_確定` 付きファイル名を付け、確定フォルダに保存する

### ファイルを探すとき

- 種別がわかっていれば、対応するフォルダを開く（例: 議事録 → `01_会議・議事録`）
- 確定済みかどうかで探す場所が分かれる: フォルダ直下 = 作業中、確定フォルダ = 確定済み
- 種別がわからなければ、Google ドライブの検索機能で検索する。命名規則に沿った名前が付いていれば、種別コードやキーワードで検索できる

### 種別コードを追加したいとき

1. Google ドライブに対応するフォルダ（と確定サブフォルダ）を作成する
2. Gem の指示に新しい種別コードと保存先フォルダを追加する

## Gem を使わずに運用する場合

Gem によるファイル名の提案は便利だが、必須ではない。命名規則を覚えている担当者は Gem を使わずに直接ファイル名を付けてもよい。

Gem の利用を強制せず、「命名規則を覚えていない人、迷った人は Gem に聞く」という位置づけにすると、導入のハードルが下がる。

## 導入効果

| 項目 | 導入前 | 導入後 |
|------|--------|--------|
| ファイル名の決定 | 各自が自由に命名（バラバラ） | Gem が命名規則に沿って提案 |
| 保存先の判断 | フォルダ構成を覚える必要がある | Gem が保存先も提案 |
| ファイルの検索 | 名前が統一されず検索しにくい | 種別コード＋キーワードで即検索 |
| 最新版の判別 | v1, v2, 最終版が乱立 | ファイルは常に1つ（履歴で過去版を確認） |
| 確定状況の把握 | 確定済みか作業中か判別できない | フォルダ（直下 / 確定）と形式（PDF）で一目瞭然 |
| 命名規則の定着 | ルールを覚えない・守らない | Gem が提案するので覚える必要なし |
| 新任者の対応 | フォルダ構成の説明が必要 | Gem に聞けば保存できる |

## 応用例

- **年度末のアーカイブ**: 年度が替わったら、各フォルダ内の確定済み文書を年度別サブフォルダに移動する
- **部門横断の共有文書**: 複数部署で共通の命名規則と Gem を使うことで、部署間でもファイルが見つけやすくなる

## 注意点

- 種別コードを追加・変更した場合は、Gem の指示も必ず更新する。Gem の指示が古いまま放置されると、存在しないフォルダを案内してしまう
- 命名規則に従わないファイルが「99_その他」や各フォルダに混在した場合は、定期的に整理する
- 確定フォルダ内の PDF は原則として編集しない。改訂が必要な場合は、作業文書として新しいファイルを作成する
